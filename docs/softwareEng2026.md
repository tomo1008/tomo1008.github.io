ソフトウェア工学まとめ

第1回
ソフトウェア工学の難しいところとして、実務から体系化された知見、知見と実践の乖離などがある。ソフトウェア開発に携わった経験がないとイメージが湧かないことや知見を理解しても実践することが難しいことが理由。個々人のソフトウェア工学の知識だけでは不十分であるからチーム全体で共有する必要がある。
ＡＩAgentによるソフトウェア開発があるが要件定義、バージョン管理、テストが不可欠である。また、監督者としての能力が求められる。

第2回
ソフトウェアの定義は、実行されることによって必要な特性、機能、性能を提供する命令語、プログラムが適切に情報を扱うことを可能とするデータ構造、プログラム 操作や使用法を記述した情報の三つがあげられる。ハードウェアと異なり劣化しないが、悪化する。ソフトウェアの特徴は、新しい環境や技術のニーズを満たすこと、新しいビジネス要求を実現するために強化されること、より近代的なシステムやデータベースと相互運用するため拡張されること、進化し続ける環境で実行可能となるよう、再構築されることである。
ソフトウェア工学ゴールは、進化論に基づく （ソフトウェアシステムが変化し続ける状況に対応する）方法論を考え出すことである。 新しいソフトウェアが古いシステムから構築され、すべては互換性を持ち、他システムと相互運用すること。
Hookerの7原則
1　システムが存在する唯一の理由は、ユーザに価値を提供するため
2　全ての設計は可能な限りシンプルであるべき
3　ソフトウェアプロジェクトの成功には明確なビジョンが不可欠である
4　あなたのやっていることを他人が理解しなければならないということを常に考えながら仕様を決め、設計し、ドキュメントを書き、実装するべきである。
5　コンピュータ環境の仕様変更に対応すべく、一般的な問題を解決するシステムを構築して、可能な限り多くの選択肢を与えられるよう準備しておく。
6　コストを低減し、再利用するコンポーネントとそれを取り込むシステムの両方の価値を高める。
7　行動を起こす前に簡潔明瞭な考えをもつことは、必ずと行ってよいほどよい結果を生む
ソフトウェア工学は品質、コスト、納期の最適なバランスを実現するための手法、方法論。ソフトウェア工学は、ソフトウェアの大規模化と複雑さ、IT人材の不足、社会性と求められるミッション、不確実性の増大に対応するために必要である。
ソフトウェア開発のコストには、財務コスト、資源コスト、人件コスト、処理コスト、機会コスト、社会コストがある。

第3回
ソフトウェアライフサイクルについて
まず要件定義から始まる。要件定義とは、実現する機能・実現しない機能を明確にすることである。要件定義の意義として、プロジェクト関係者のバイブル、社内外への宣言と協力依頼、RFPの作成など多岐にわたる。
要件定義書から設計書へ移る。ここでコーディングができるレベルの仕様書に落とし込む。設計書に従ってシステム構築をする。自社開発と外部委託の2種類がある。外部委託の場合、オフショア開発といった賃金の安い海外企業に外部委託する手法がある。注意として、文化、言語の違いなどがあげられる。
テストとデバッグをする際にバグのないソフトウェアはないと考えるべき。あらかじめテストしバグを早期に発見・対処することが理想。すべてのすることは工数的に不可能であるからどの程度テストをするかなどを決めておく必要がある。
運用、保守では稼働後のデータをプロアクティブに活用し、ソフトウェア以外の要因も考慮する必要がある。開発者はプロジェクトから離れている可能性があることに注意。
複数人でのソフトウェア開発には非同期コミュニケーションが重要で、そのためのドキュメンテーション。ほとんどが英語であるため英語に慣れる必要がある。

第4回
プロジェクトについて
プロジェクトとは有期性と独自性があるもの
フォアキャスティングとはインプット視点で今の立ち位置から順に目標に進んでいく。
バックキャスティングとはアウトプット視点で定めた目標に向けて何が必要かを計画し達成に向かう。
ソフトウェアの評価方法は三つある。ステップ数、オブジェクト容量、FP法がある。組み込み型ソフトウェアはサイズが小さいほどよいためオブジェクト容量での評価。FP法はプログラミングフェーズに入る前にユーザー要件が定まり、必要な機能が見えてきた段階でシステム規模を概算することができる。
使い勝手でも評価することができる。
開発プロセスには、ウォーターフォール型開発プロセス、スパイラルモデル、反復型開発プロセス、アジャイルプロセス、リスク駆動形開発プロセスがある。
ウォーターフォール型開発プロセス
メリット　進捗管理が容易、成果物が明確
デメリット　後工程にしわ寄せが集中というリスクあり
スパイラルモデル
メリット　プログラム開発を小さなフェーズに分割し、フェーズごとにプロトタイプによるデモンストレーションからのフィードバックをすることができる。
デメリット　プロトタイプ作成に想定外の作業量が発生するリスクあり
反復型開発プロセス
メリット　部分的に完成させていくので、顧客の要求を取り入れやすい。また、部分的な納品が可能。
デメリット
分割のための作業や管理業務が増える
全体像が見えづらい
一括して稼働するシステムは分割しても意味がなくなってしまう
アジャイルプロセス
メリット　
変化に対応できること
シンプルな設計
速いサイクルで頻繫にな納品
デメリット　
開発スキルが高くないと採用することが難しい
コストが高い
一長一短で場面や要求によるためハイブリッド型が多くある。

第5，6回
WBSとはプロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したもの
メリット
スコープが明確になる（やらないことが明確になる）
作業を洗い出す（やるべき作業が明確になる）
全体管理と作業計画が明確化される
プロジェクト実施時のWBSに則り実行するのみ
WBSの作り方
1　スコープを明確にする
2　大きな作業をグルーピングする
3　グルーピングした作業の相互関連を考える
4　各グループの作業を洗い出す

第8回
バージョン管理をすることでファイルを誰が、いつ、どのように変更したかを管理する。管理することで多数のメンバーとも共同でコード開発ができる。集中管理型や分散管理型などがある。
集中管理型のデメリットは、同時編集をするとコンフリクトが発生しやすいことや同期に時間がかかること。
分散管理型のメリットは、リモートリポジトリへのアクセス頻度が少なく、障害に頑健。
gitは分散管理型のバージョン管理システムである。
コミットは変更の記録でいつ、だれが、なにを、どのようなことをしたかがわかる。メッセージを残すこともできる。
レポジトリはgitが管理するプロジェクトをのフォルダのことである。
ローカルレポジトリでの流れ
ファイルの変更を保存してそのファイルを登録する。
そのファイルをコミットをすることで変更を登録する。コミットは原則、変更、削除できない。
リモートレポジトリの流れ
ローカルレポジトリでコミットされた変更をリモートレポジトリへ反映するコマンドが git push
リモートレポジトリにある内容をローカルレポジトリに反映するコマンドがgit pull
ブランチを使うことで、作業を枝分かれさせることができるため、共同作業、並行作業が可能。

第12、13回
Continuous Integration(CI)
―コード変更を共有リポジトリに頻繁に統合するプロセス
-自動テストとビルドを定期的に実行し、バグの早期発見と修
正を可能に
-CIの自動化により開発のスムーズな進行を促進
Continuous Delivery (CD)
-コード変更をテスト環境や本番環境に自動的にデプロイする
プロセス
-自動デプロイメントを組み込み、手動操作によるデプロイの
必要性を排除
-CDにより、ユーザーフィードバックを迅速に反映可能

CI/CDパイプラインの基本的なステップ
1. ソース: コード変更をトリガーにワークフローを起動
2. ビルド: ソースコードをコンパイル
3. テスト: 自動テストを実行
4. デプロイ: テスト済みのコードを本番環境にデプロイ
5. 検証: デプロイされたアプリケーションの動作確認
6. モニタリング : 本番環境での継続的な監視
CIにflake8を設定することで自動的に判定してくれるようにすることが可能
